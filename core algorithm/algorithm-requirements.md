# 三角网格圈地博弈游戏 — 核心算法需求文档（重构版）

> **版本说明**：本文档基于"双层架构重构"后的最终实现，全面替换旧版几何射线法文档。

---

## 一、基本设置

### 1. 棋盘网格

- **坐标系**：9×9 等边三角形网格。
- **坐标分布**：第 y 行有 `9 - y` 个点，x 从 0 到 `8 - y`。
  - 第 0 行 9 个点：`(0,0)` → `(8,0)`
  - 第 1 行 8 个点：`(0,1)` → `(7,1)`
  - … 依次递减，第 8 行 1 个点：`(0,8)`
- **屏幕坐标映射**（线性双射，保持拓扑）：
  ```
  screen_x = grid_start_x + x * cell_size + y * cell_size // 2
  screen_y = grid_start_y + y * cell_size * 0.866
  ```

### 2. 六向邻接定义

在三角形网格中，每个点 `(x, y)` 的六个合法相邻点为（顺时针，E→SE→SW→W→NW→NE）：

```
(x+1, y),  (x, y+1),  (x-1, y+1),
(x-1, y),  (x, y-1),  (x+1, y-1)
```

仅包含在 `grid` 字典中存在的点（即满足 `0 ≤ x` 且 `x + y ≤ 8`）。

### 3. 实体与状态

| 状态常量 | 含义 |
|---|---|
| `EMPTY` | 空白点 |
| `BLACK_NODE` | 蓝方节点（主动落子，防线支柱） |
| `BLACK_LINE` | 蓝方线点（节点间自动生成，可牺牲） |
| `WHITE_NODE` | 红方节点 |
| `WHITE_LINE` | 红方线点 |

### 4. 初始状态

- 蓝方（BLACK）出生节点：`(0, 0)`
- 红方（WHITE）出生节点：`(8, 0)`
- 保护区：对手出生节点的所有六向相邻点，己方不得落子。

## 二、双层架构：逻辑拓扑层与网格物理层分离

### 2.1 架构总览

本游戏采用**双层数据模型**，将"渲染表示"与"连通性判断"彻底解耦：

| 层次 | 数据结构 | 职责 |
|---|---|---|
| **网格物理层** | `self.grid: dict[(x,y) → PointState]` | 驱动屏幕渲染，记录每个点的当前归属状态 |
| **逻辑拓扑层** | `self.black_edges / white_edges: set[frozenset]` | 记录已确立的节点间显式边，驱动连通性判断（BFS） |

### 2.2 逻辑拓扑层（adj_list / edges）

- **数据结构**：`black_edges` 和 `white_edges` 均为 `set[frozenset({node_a, node_b})]`，每个元素代表一条已确立的节点间连接边。
- **写入时机**：
  1. 落子成功 `_add_node` 后，对每条新建立的连线执行 `edges.add(frozenset({node1, node2}))`。
  2. 重新缝合防线 `_reconnect_player_nodes` 后，同步写入新边。
- **清理时机**：
  1. 攻击发生后（Step 1.5），调用 `_cleanup_broken_edges`：扫描边集合，移除线段上已有点不再归属该玩家的断裂边。
  2. 孤立节点被删除时（Step 2），调用 `_remove_node_edges`：移除所有经过该节点的边。
- **连通性查询**：`_is_connected_to_initial(pos, player)` 从边集合构建临时邻接表，执行 BFS，判断节点是否仍连通到出生点，**完全不依赖 `self.grid`**。

### 2.3 网格物理层（grid）

- **数据结构**：`self.grid: dict[(x,y) → PointState]`，全局唯一。
- **职责**：仅用于渲染和落子合法性检查（避障、敌方占领判断）。
- **与逻辑层的关系**：渲染层是逻辑层的视觉投影；连通性判断永远查询逻辑层（edges），而非扫描 grid 点集。

---

## 三、基础连线规则

任意两节点可建立直线连接，当且仅当满足以下三方向之一：

1. **水平**：`y1 == y2`（同行）
2. **竖直**：`x1 == x2`（同列）
3. **对角**：`x1 + y1 == x2 + y2`（同斜率对角线）

连线中间路径上不得有任何敌方点（`_can_connect_with_blocking`）。

---

## 四、游戏流程与交互

每回合严格按序执行：

1. **落子扩张**：在空白点或敌方线点落子，落点须与己方至少一个节点可直线连通（无阻挡）。
2. **自动连线**：新节点与所有可达己方节点自动建立连线，沿途空白点转为己方线点，同步写入 `edges`。
3. **三点限制**：不得主动形成三个己方节点两两相邻的锐角阵型（进攻落子豁免此规则）。
4. **保护区限制**：不得落子于对手出生节点的六向相邻点。

---

## 五、进攻与结算机制

当新节点落在**敌方线点**上时触发攻击，严格按以下四步结算：

### Step 1：级联删除线点

以落子点为中心，向六个方向射线扫描。若相邻点是敌方线点，则沿该方向持续清除敌方线点，直到：
- 遇到敌方**节点**：保留节点，停止（此方向确认为同一线段，批量删除已收集的线点）。
- 遇到空白点或边界：取消删除（该方向线点不属于被切断的线段，避免误伤垂直线段）。

### Step 1.5：逻辑层同步清理

调用 `_cleanup_broken_edges(opponent)`：从对手的边集合中移除因线点被删除而断裂的边（线段上有格点已不再归属对手）。

### Step 2：BFS 飞地检测 → 删除孤立节点

遍历对手所有幸存节点，调用 `_is_connected_to_initial`（查询逻辑层 edges，执行 BFS）：
- 无法连通出生点的节点 → 判定为**飞地**，置为 `EMPTY`。
- 同步调用 `_remove_node_edges` 从边集合中移除所有经过该节点的边。

### Step 3：孤立线点清理

遍历对手所有线点，检查其是否仍处于某条两端节点均幸存的完整线段上（所有中间点仍归属对手）。不满足条件的线点一律置为 `EMPTY`。

### Step 4：重新缝合防线

调用 `_reconnect_player_nodes`（对攻守双方各执行一次）：扫描所有己方节点对，若满足连线规则且无阻挡，则在中间空白点建立线点，并写入 `edges`。

## 六、核心领土算法（三阶段流水线）

### 阶段一：右手摸墙法 — 获取外轮廓

**方法**：`_get_outer_contour(player)`

**目标**：沿己方棋子（节点+线点）的物理外边缘，追踪出一条闭合轮廓路径 `current_poly`。

**算法流程**：

1. **起点**：从所有己方点中选 x 最小（相同取 y 最小）的点作为 `start`。
2. **初始朝向**：假定探针从正西方（方向 3 = `(-1, 0)` 的反向）进入起点。
3. **右手摸墙迭代**：
   - 从 `(backtrack + 1) % 6` 开始顺时针扫描 6 个方向，取第一个属于己方点的方向为 `out_dir`。
   - 移动到下一格，更新 `backtrack = (out_dir + 3) % 6`（反向）。
   - 将经过的点追加到 `contour`。
4. **闭合终止**：当探针回到 `start` 且准备迈出方向与首步完全相同时，停止（**首尾不重复**，隐式闭合）。
5. **单点/孤立情形**：若仅有一个点或找不到任何出路，直接返回该点的单元素列表。

**输出**：`List[Tuple[int, int]]`，隐式闭合（首尾不重复），列表长度即为**轮廓周长（跳数）**。

---

### 阶段二：动态贪心修剪 — 皮筋收紧算法

**方法**：`_compute_inner_hull(player)` 的 Step 2

**目标**：在 `current_poly` 内持续寻找"捷径"，将轮廓向内勒紧，切除包裹在外壳的冗余线点。

**循环结构**（`while True`）：

每轮循环中：

1. **当前基准**：记录 `cur_perim = len(current_poly)`，调用泛洪法得到 `cur_area = len(_get_covered_points(current_poly))`。
2. **遍历锚点对**：正序枚举起点 `i`，倒序枚举终点 `j`（`j > i+1`），调用 `_get_all_shortest_grid_paths(poly[i], poly[j], enemies)` 获取所有等长最短格点路径（上限 100 条，避开所有敌方点）。
3. **生成双向候选**：对每条 `path`，生成两个候选多边形：
   - **方案 A（替换内弧）**：`poly[:i] + path + poly[j+1:]`
   - **方案 B（替换外弧）**：`poly[i:j+1] + path[::-1][1:-1]`
4. **去除相邻重复顶点**（防止周长虚增）。
5. **独立校验**（每个候选独立参加）：
   - **周长筛选**：`cand_perim <= best_cand_perim`，否则直接淘汰。
   - **泛洪法计算覆盖领土**：`covered = _get_covered_points(cand)`，`cand_area = len(covered)`。
   - **面积筛选**：若周长相等但 `cand_area >= best_cand_area`，淘汰。
   - **核心资产保护**：`friendly_nodes.issubset(covered)`（所有节点必须在领土内，允许丢弃冗余线点）。
   - **避障测试**：`not any(e in covered for e in enemies)`（领土内不得包含任何敌方点）。
6. **全局最优追踪**：通过全轮遍历后取最优候选（优先周长最短，次之领土点数最少），更新 `current_poly`。
7. **收敛判断**：若本轮无任何候选通过，退出循环。

---

### 阶段三：泛洪法领土判定

**方法**：`_get_covered_points(polygon)`

**目标**：确定轮廓多边形真正围住的网格点集合，**完全不依赖几何射线法**，天然支持自交多边形。

**算法流程**：

1. **建立防水墙**：将 `polygon` 中所有点加入 `wall_set`。
2. **边缘注水**：从棋盘三条物理边缘（`x==0`、`y==0`、`x+y==8`）的所有**非墙**点开始，加入 BFS 队列。
3. **BFS 蔓延**：水流向六向相邻格点扩散，遇到 `wall_set` 中的点则停止（墙体不可穿透）。
4. **计算领土**：`领土点集 = 全图所有格点 - 被水淹没的点集`（含轮廓墙体本身）。

**输出**：`set[Tuple[int, int]]`，即真实覆盖领土的所有格点。

---

### 完整输出

`_compute_inner_hull(player)` 返回：
- `screen_polygon: List[Tuple[int, int]]`：用于 Pygame 渲染的屏幕坐标轮廓（首尾相接，闭合）。
- `area: int`：泛洪法确定的领土格点总数（代替几何面积参与大小比较）。

---

## 七、胜利条件

- **结束**：双方均无合法棋可走，或双方连续各跳过一次（`consecutive_skips >= 2`）。
- **胜利**：领土格点数（`area`）更大的一方获胜；相等则平局。

---

## 八、关键设计决策汇总

| 决策点 | 旧方案 | 新方案（当前实现） |
|---|---|---|
| 领土内外判断 | 几何射线法（Ray-Casting） | 网格泛洪法（Flood-Fill BFS） |
| 面积计算 | 斜坐标 Shoelace 公式 | 领土格点总数（离散计数） |
| 连通性判断 | 扫描 grid 点集 | 查询逻辑边集合（edges）BFS |
| 外轮廓获取 | `_compute_inner_hull` 内联实现 | 独立封装 `_get_outer_contour`，主函数直接调用 |
| 自交多边形处理 | 射线法产生拓扑误判 | 泛洪法天然兼容，无拓扑约束 |
| 节点边界点区分 | `_is_on_segment` 辅助判断 | 领土集合直接包含轮廓点，无需区分 |
