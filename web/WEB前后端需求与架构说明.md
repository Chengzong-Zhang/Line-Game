# LIFELINE Web 前后端需求与架构说明

## 1. 文档目的

本文档用于重新梳理 LIFELINE Web 端当前的真实需求、前后端职责边界、模块结构与后续扩展入口。内容依据以下资料整理：

- 仓库根目录 `README.md`
- `web/新的需求.md`
- 当前在线运行链路对应源码
- 现有 `web/web前端` 与 `web/web后端` 目录结构

本文档优先描述“当前真实实现”和“当前应继续沿用的架构方向”，并明确区分历史方案与实验性分支，避免后续开发时继续混用旧认知。

## 2. 项目目标与范围

LIFELINE 是一个基于三角网格的圈地类棋盘游戏网页端，目标是在同一套规则引擎下，同时支持本地对战与联机对战，并提供可在桌面端和移动端运行的统一界面。

当前范围如下：

- 支持双人和三人对局
- 支持边长 `5` 到 `15` 的三角棋盘
- 支持本地模式与基于 WebSocket 的联机模式
- 支持账号注册、登录与带鉴权的联机连接
- 支持中英文界面切换
- 支持房间创建、加入、准备、开始、重置和离开
- 支持断线后的会话恢复与重新进入房间
- 支持以棋盘为核心的极简布局与折叠式信息仓

## 3. 当前真实架构原则

### 3.1 核心原则

当前 Web 端采用“前端负责游戏规则，后端负责房间与同步”的架构。

- 前端 `GameEngine` 负责棋盘模型、合法落子、回合推进、领地统计、终局判断等规则逻辑
- 前端 `Renderer` 负责 Canvas 绘制
- 前端 `GameController` 负责把规则层、渲染层、交互层和网络层衔接起来
- 后端 `server.py` 负责账号系统、静态资源、房间状态、玩家连接、操作转发、准备机制、重置投票和断线重连
- 后端不负责三角棋盘几何规则判定，也不负责领地和胜负裁决

### 3.2 当前真实入口链路

当前真实前端入口链路是：

`index.html -> main.js -> OnlineApp.js`

当前真实后端主服务是：

`web/web后端/server.py`

### 3.3 历史或实验性路径

仓库中存在 `web/web后端/game_router.py`。该文件体现的是“后端作为规则单一真源”的实验性方向，与当前实际运行架构不一致。

因此后续开发默认遵循以下结论：

- 当前主线以 `server.py` 为准
- 当前规则计算以浏览器端 `GameEngine.js` 为准
- `game_router.py` 可视为实验性或历史性探索文件，不应被误认为当前在线主链路

## 4. 功能需求重写

### 4.1 基础对局能力

- 玩家可发起本地对局
- 玩家可创建或加入联机房间
- 对局支持 `2` 人或 `3` 人
- 棋盘边长支持 `5` 到 `15`
- 同一套规则引擎同时服务本地与联机模式
- 三人模式下需要完整支持第三方颜色、第三方回合与第三方领地统计

### 4.2 本地模式需求

- 本地模式无需账号即可开始
- 用户可选择人数、边长和先手
- 本地模式下支持正常落子、跳过回合和重置对局
- 棋盘状态与本地操作始终围绕棋盘展开，不额外依赖联机服务

### 4.3 联机模式需求

- 联机前需要完成账号注册或登录
- 登录成功后可建立带 Token 的 WebSocket 连接
- 用户可创建房间或加入已有房间
- 房间内需要同步玩家身份、颜色、房主、设置、准备状态、倒计时和操作记录
- 联机模式下落子、跳过和重置等动作需要广播到房间成员
- 玩家断线后可凭本地持久化的会话信息尝试恢复进入原房间

### 4.4 房间与场间逻辑需求

- 房主可创建房间
- 其他玩家可通过房间号加入
- 房间支持展示当前模式为本地还是联机，以及当前房间号
- 房主可以修改房间设置
- 房主可以修改先手方
- 一旦房间设置或先手发生变化，所有玩家的准备状态需要被重置
- 对局开始前使用“准备”机制而不是单向开始按钮
- 所有需要开始对局的玩家都准备完成后，进入倒计时再开局
- 联机重置采用全员确认机制
- 玩家可主动离开房间
- 玩家断开连接后，系统应在一段有效时间内保留房间上下文，便于恢复

### 4.5 当前 UI 与信息结构需求

当前 UI 不再追求把所有信息平铺在首屏，而是强调“先看到棋盘，再按需展开信息仓”。

#### 页面主目标

- 用户打开页面后，应优先看到页面标题与中央棋盘
- 棋盘必须完整显示，不能在手机端发生关键区域溢出
- 棋盘区域应保持稳定尺寸策略，避免因视口高度抖动导致卡顿

#### 页面信息层次

- 顶部展示品牌标题
- 中英文切换时，标题需同步切换为 `生命线` 或 `LIFELINE`
- 棋盘下方保留极简对弈模块，仅展示当前回合和本地操作按钮
- 棋盘旁保留两个折叠仓

#### 棋盘仓

棋盘仓默认折叠，仅显示图标、名称和必要摘要；展开后纵向展示：

- 对局设置
- 棋盘状态

折叠态至少保留：

- 当前人数
- 当前边长

#### 联机仓

联机仓默认折叠，仅显示图标、名称和必要摘要；展开后纵向展示：

- 联机对局
- 账号系统

折叠态至少保留：

- `本地` 或当前房间号

#### 其他 UI 约束

- 不再保留冗余播报区
- 不在棋盘区域内放置额外说明性长文本
- 折叠仓应默认闭合，未点击时只展示目录级信息
- 页面滚动方向以竖向为主
- 移动端要优先保证棋盘完整展示和基本操作可见

### 4.6 国际化需求

- 前端需提供中英文文案
- 品牌标题、按钮、状态、错误提示和比分文本都应可本地化
- 新增界面模块时，文案优先补充到统一文案文件中，而不是散落在组件内硬编码

## 5. 前端架构说明

前端目录位于 `web/web前端`，当前核心结构如下。

### 5.1 `index.html`

职责：

- 作为页面壳文件
- 挂载应用根节点
- 加载 `main.js`
- 通过版本号控制静态资源缓存

### 5.2 `main.js`

职责：

- 负责前端启动
- 检查并加载 Vue 运行时
- 动态导入 `OnlineApp.js`
- 应用挂载失败时给出启动错误界面

说明：

- 这是当前真实前端入口
- 静态资源版本号也在这里集中控制

### 5.3 `OnlineApp.js`

职责：

- 作为当前主应用编排层
- 组织页面整体布局与 Vue 组件
- 管理本地模式与联机模式的上层流程
- 衔接 `GameController`、`NetworkManager`、共享状态和国际化模块
- 承担大部分页面交互逻辑、房间操作逻辑和 UI 展示逻辑

当前特点：

- 仍然偏大，属于“主应用文件 + 内联组件集合”的结构
- 是后续继续拆分前端组件时的首要切分入口

### 5.4 `OnlineAppState.js`

职责：

- 维护默认对局设置
- 提供人数与边长配置项
- 管理会话存储 Key
- 管理认证信息、本地会话和默认状态对象
- 提供持久化与恢复工具函数

说明：

- 这是页面共享状态工具层
- 当前会话恢复的是“上下文信息”，不是自动恢复一个已存在的浏览器连接

### 5.5 `OnlineAppI18n.js`

职责：

- 维护中英文文案
- 统一格式化标题、比分、状态和错误提示
- 为整个应用提供语言切换支撑

说明：

- 这是当前 UI 文案的单一真源
- 新增页面文本应优先收敛到这里

### 5.6 `GameController.js`

职责：

- 连接规则层、渲染层与交互层
- 接收页面操作并驱动引擎
- 接收网络同步动作并转交给本地控制流程

说明：

- 这是对局控制中枢
- 一般不直接承载复杂 UI，只负责对局控制编排

### 5.7 `GameEngine.js`

职责：

- 维护棋盘数据结构
- 判定合法落子
- 推进玩家回合
- 统计领地与比分
- 判定终局状态
- 对外提供可用于渲染和同步的快照

说明：

- 这是当前规则计算真源
- 用户明确要求不要随意改动领地计算、删点计算等规则算法

### 5.8 `Renderer.js`

职责：

- 基于 Canvas 渲染棋盘、节点、连线、领地和高亮
- 处理画布尺寸、像素比和重绘策略
- 通过渲染缓存与批量绘制承担性能优化职责

说明：

- 后续若继续解决大棋盘卡顿，应优先在这一层优化
- 不应通过修改规则算法去掩盖渲染层性能问题

### 5.9 `NetworkManager.js`

职责：

- 封装 WebSocket 连接
- 负责鉴权 Token 透传
- 发送建房、入房、落子、跳过、重置、准备、房间设置等消息
- 监听服务端广播事件
- 管理心跳、连接关闭与错误事件

说明：

- 当前采用“单一 WebSocket 同时承载请求响应和广播事件”的模式
- 上层只需订阅事件并调用发送方法，无需直接操作原生 socket

### 5.10 `styles.css`

职责：

- 提供页面布局、品牌样式、棋盘区、折叠仓、对弈模块与移动端适配样式
- 管理界面的固定尺寸策略与响应式展示

### 5.11 `smoke-test.html`

职责：

- 用于快速验证基础前端交互与页面加载是否正常

## 6. 后端架构说明

后端目录位于 `web/web后端`。

### 6.1 `server.py`

这是当前真实在线主服务，职责如下：

- 提供 FastAPI 应用入口
- 提供注册与登录接口
- 提供 JWT 鉴权
- 提供静态资源托管
- 提供 WebSocket 房间服务
- 管理房间生命周期
- 管理准备状态、倒计时、重置投票与断线恢复
- 转发联机动作消息

明确不负责的内容：

- 不负责三角棋盘规则判定
- 不负责领地与终局计算
- 不负责替代前端引擎成为游戏规则真源

### 6.2 `database.py`

职责：

- 初始化数据库连接
- 提供会话工厂与表初始化能力

### 6.3 `models.py`

当前核心模型：

- `User`

职责：

- 存储用户 ID、用户名和密码哈希

### 6.4 房间运行模型

`server.py` 当前以进程内内存结构管理房间，关键对象包括：

- `PlayerSession`
- `Room`
- `ConnectionManager`

其中：

- `PlayerSession` 表示玩家身份、颜色、连接状态和最近心跳时间
- `Room` 表示房间配置、玩家列表、动作记录、准备状态、重置投票和倒计时任务
- `ConnectionManager` 负责整个连接生命周期和所有房间操作分发

### 6.5 当前后端关键常量

以 `server.py` 当前实现为准：

- 房间最大人数上限由房间设置决定，系统总体支持到 `3` 人
- 房间号长度为 `4`
- 房间超时清理时间为 `300` 秒
- 心跳超时为 `35` 秒
- 心跳巡检周期为 `5` 秒
- 准备完成后的开始倒计时为 `3` 秒
- 可用棋盘边长范围为 `5` 到 `15`

### 6.6 认证机制

- 注册与登录通过 HTTP 接口完成
- 密码使用 `bcrypt + sha256` 方式存储与校验
- 登录后签发 JWT
- WebSocket 连接通过 URL 参数附带 Token
- 服务端解析 Token 后识别用户名并允许进入联机流程

## 7. 关键数据流

### 7.1 本地模式数据流

1. 用户在前端选择人数、边长和先手
2. `OnlineApp.js` 调用 `GameController`
3. `GameController` 初始化 `GameEngine`
4. `Renderer` 根据引擎快照绘制棋盘
5. 用户点击棋盘时，由控制层把输入交给引擎计算并刷新渲染

本流程不依赖后端。

### 7.2 联机建房与入房数据流

1. 用户注册或登录
2. 前端获得 Token 并持久化认证状态
3. `NetworkManager` 使用 Token 建立 WebSocket 连接
4. 房主发送 `create_room`
5. 服务端创建房间并返回 `ROOM_CREATED`
6. 其他玩家发送 `join_room`
7. 服务端返回 `ROOM_JOINED` 并广播最新房间状态
8. 前端根据房间状态初始化或更新本地对局展示

### 7.3 联机对局数据流

1. 某个玩家在前端点击合法节点
2. `GameController` 处理本地交互
3. `NetworkManager` 发送 `player_move`
4. 服务端把动作转发给房间其他玩家
5. 其他玩家前端收到事件后，用同一套前端规则链路回放这次动作

联机模式下，服务端负责“同步动作”，前端负责“解释动作并更新棋局”。

### 7.4 准备与开始数据流

1. 房间成员点击准备
2. 前端发送 `player_ready`
3. 服务端更新房间准备状态并广播
4. 当所有需要开始对局的玩家都准备完成后，服务端开始倒计时
5. 倒计时结束后，房间进入已开始状态
6. 前端据此刷新对局 UI

### 7.5 房间设置变更数据流

1. 房主修改人数、边长或先手
2. 前端发送 `update_room_settings` 或 `update_start_player`
3. 服务端更新房间设置
4. 服务端清空准备状态并广播最新房间信息
5. 各客户端据此重置准备展示并等待再次确认开始

### 7.6 联机重置数据流

1. 某玩家发起重置
2. 前端发送 `player_reset`
3. 服务端记录该玩家投票
4. 服务端广播当前重置投票状态
5. 当所有房间成员都同意后，服务端确认本轮重置成立
6. 前端基于最新设置重新开始一局

### 7.7 断线恢复数据流

1. 客户端断开连接
2. 服务端暂时保留玩家会话与房间上下文
3. 前端从本地存储恢复认证与最近房间信息
4. 用户重新连接并尝试加入原房间
5. 服务端根据玩家身份恢复其在房间中的位置

## 8. WebSocket 协议摘要

### 8.1 前端发送的主要消息

以 `NetworkManager.js` 和 `server.py` 当前实现为准，前端主要发送：

- `create_room`
- `join_room`
- `player_move`
- `player_skip`
- `player_reset`
- `player_ready`
- `update_room_settings`
- `update_start_player`
- `player_leave`
- `ping`

### 8.2 前端监听的主要服务端事件

以 `NetworkManager.js` 当前实现为准，前端主要监听：

- `ROOM_CREATED`
- `ROOM_JOINED`
- `ROOM_STATE`
- `ROOM_COUNTDOWN`
- `ROOM_READY`
- `OPPONENT_MOVE`
- `TURN_SKIPPED`
- `RESET_STATUS`
- `MATCH_RESET`
- `PLAYER_LEFT`
- `PONG`
- `ERROR`

### 8.3 协议设计特征

- 同一条 WebSocket 同时承担请求响应与广播职责
- 客户端需要同时处理点对点返回和房间广播
- 心跳机制用于检测连接是否仍然有效

## 9. 修改入口建议

当后续需要继续开发时，建议按以下顺序理解和修改：

1. `web/web前端/main.js`
2. `web/web前端/OnlineApp.js`
3. `web/web前端/OnlineAppState.js`
4. `web/web前端/OnlineAppI18n.js`
5. `web/web前端/GameController.js`
6. `web/web前端/GameEngine.js`
7. `web/web前端/Renderer.js`
8. `web/web前端/NetworkManager.js`
9. `web/web后端/server.py`

如果目标是：

- 调整 UI 或交互：优先看 `OnlineApp.js` 与 `styles.css`
- 调整文案或语言：优先看 `OnlineAppI18n.js`
- 调整默认配置或本地存储：优先看 `OnlineAppState.js`
- 调整规则：优先看 `GameEngine.js`
- 解决大棋盘卡顿：优先看 `Renderer.js`
- 调整联机协议或房间逻辑：优先看 `NetworkManager.js` 和 `server.py`

## 10. 当前开发约束

为了避免架构继续发散，后续开发建议遵守以下约束：

- 不要把规则真源从前端悄悄迁移到后端，除非有明确的整体架构升级方案
- 不要在 `OnlineApp.js` 中继续堆叠大量硬编码文案
- 不要通过修改规则算法来掩盖渲染性能问题
- 不要让移动端适配重新回到“所有模块平铺首屏”的状态
- 新增联机能力时，应先明确是“服务端权威同步”还是“前端规则回放”，避免混搭

## 11. 验证清单

每次涉及 Web 端调整后，至少验证以下项目：

- 本地模式下可正常开始双人局和三人局
- 边长 `5`、`10`、`15` 下棋盘都能完整显示
- 移动端下棋盘不溢出，且页面不卡顿到无法交互
- 当前回合、本地按钮与棋盘状态展示正常
- 棋盘仓默认折叠，展开后顺序为“对局设置 -> 棋盘状态”
- 联机仓默认折叠，展开后顺序为“联机对局 -> 账号系统”
- 折叠态摘要能正确显示人数、边长、房间号或本地状态
- 中文标题显示为“生命线”，英文标题显示为 `LIFELINE`
- 注册、登录、建房、入房、准备、开始、落子同步和离房流程正常
- 房主修改设置后，房间准备状态会被正确重置
- 联机重置仍然是全员确认机制
- 断线后重新连接能尽量恢复原房间上下文

## 12. 一句话结论

LIFELINE 当前 Web 架构的正确理解是：

前端负责规则与渲染，后端负责账号、房间和同步；当前主线文件是 `main.js + OnlineApp.js + server.py`，后续重构应围绕这条真实链路持续收敛，而不是回到旧入口或混入实验性后端规则分支。
