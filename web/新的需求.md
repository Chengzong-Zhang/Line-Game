# Prompt: 15x15 棋盘游戏性能终极优化方案（双层架构与脏矩形思路）

## 1. 现状分析
当前 15x15 棋盘在移动端仍有卡顿。虽然已实现了批量绘制和减少 Reflow，但目前的渲染逻辑仍是“全量重绘”：即使只落一个子，也会清空整个画布并重新绘制几百条线和上百个点。在高 DPR 的移动端，这种像素填充率（Fill Rate）压力极大。

## 2. 核心目标
将渲染效率提升一个量级，确保 15 边长棋盘在移动端滚动和落子时达到 60FPS。

## 3. 开发约束（底线）
* **禁止修改逻辑算法**：不要动 `GameEngine.js` 里的领土计算、胜负判定等规则逻辑。
* **数据驱动**：渲染器通过 `snapshot` 获取数据，保持只读。

## 4. 必须实施的高级优化任务

### A. 实施双层 Canvas 架构 (Renderer.js 重构)
这是最核心的修改。请将单层 Canvas 拆分为“双层叠加”模式：
1. **静态底图层 (Background Layer)**：
   - 仅绘制：棋盘底色、三角形网格线、坐标索引。
   - 触发时机：仅在 `resize` 或 `gridSize` 改变时重绘一次。
   - 技术实现：使用一个独立的离屏 Canvas 或第二张叠加在底部的 Canvas。
2. **动态交互层 (Active Layer)**：
   - 仅绘制：棋子、合法落点高亮、最后一步光圈、悬停预览。
   - 触发时机：落子、游戏状态更新或鼠标/触控移动时。
   - 优势：每帧重绘的元素从几百个减少到几十个。

### B. 引入“按需渲染”机制 (Dirty Checking)
1. **状态比对**：在 `Renderer.render()` 执行前，对比当前 `snapshot` 与上一帧 `snapshot`。如果棋盘状态（点、线、领土）未改变，直接跳过动态层的重绘。
2. **RequestAnimationFrame 锁**：确保任何渲染调用都必须排队在 `rAF` 中，严禁在同步代码中高频触发 `render()`。

### C. 渲染细节微调 (Low-level Tweak)
1. **路径预烘焙**：对于三角形网格这种重复几何图形，尝试使用 `Path2D` 对象进行预存，减少 CPU 每次计算顶点坐标的开销。
2. **像素对齐**：确保所有绘制坐标为整数（使用 `| 0` 或 `Math.round`），避免浏览器为了处理“亚像素”而产生的抗锯齿性能损耗。
3. **强制 DPR 上限**：无论屏幕多好，将移动端的 `devicePixelRatio` 强制锁定在 `min(2, window.devicePixelRatio)`。

### D. 布局稳定性 (styles.css)
1. **彻底解耦 Viewport**：棋盘容器使用 `aspect-ratio: 4/3`，宽高使用 CSS 变量控制。
2. **禁用合成层特效**：在 `.large-board` 类下，强制 `backdrop-filter: none !important` 和 `mask-image: none !important`。
