# Prompt: 15x15 棋盘游戏前端性能深度优化（仅限渲染与布局层）

## 1. 任务背景与目标
当前项目是一个基于**三角形网格逻辑**的棋盘游戏。在 **15x15 边长**的大棋盘模式下，以及在**移动端浏览器**滚动时，页面卡顿非常严重（帧率极低）。
**核心目标：** 在不改变任何游戏逻辑的前提下，通过重构渲染链路和布局策略，使 15 边长棋盘在手机端达到丝滑运行的水平。

## 2. 诊断出的性能瓶颈
* **布局抖动 (Layout Thrashing)**：棋盘高度绑定了 `100dvh`，移动端工具栏伸缩导致频繁触发布局重算。
* **重绘冗余**：`ResizeObserver` 缺乏节流且监听逻辑重叠，导致高频触发 `resize + render`。
* **Canvas 负载过重**：
    * 移动端默认使用 3x 的 `devicePixelRatio`，导致像素处理量过大。
    * 渲染层每帧都在重新推导节点连线（$O(N^2)$ 复杂度），而非复用引擎数据。
    * 缺乏静态内容缓存，棋盘背景和网格每帧都在重复绘制。
* **合成压力**：`backdrop-filter` 和 `mask-image` 等 CSS 特效在移动端滚动时造成巨大负担。

## 3. 开发限制（重要！）
* **禁止修改核心算法**：严禁变动 `GameEngine.js` 中关于领土计算（Territory）、删点逻辑等游戏规则。
* **数据流向**：渲染器应保持只读，仅通过 `snapshot` 获取数据。
* **显示要求**：必须确保棋盘在不同屏幕尺寸下均能**完整显示**，不得超出视口。

## 4. 优化实施需求

### A. 布局与样式优化 (styles.css)
1.  **尺寸固定策略**：停止使用 `dvh` 单位。棋盘高度应改用更稳定的 `svh` 或通过 JavaScript 在初始化时计算一次固定宽高比（如 4:3），确保移动端滚动时不触发布局重排。
2.  **视觉降级**：检测到大尺寸棋盘（>= 15）或移动端环境时，自动移除 `backdrop-filter` 和 `mask-image` 特效。

### B. 渲染引擎优化 (Renderer.js)
1.  **静态/动态分层 (Layering)**：
    * 实现**静态层缓存**（离屏 Canvas 或 ImageBitmap）：将棋盘背景、网格骨架、坐标点等不变量绘制一次并缓存。
    * **动态层绘制**：每帧仅重绘棋子、高亮落点、光圈等变化内容，然后将静态缓存层叠加。
2.  **批量绘制 (Batching)**：
    * 优化渲染循环：严禁在循环内反复执行 `beginPath()`。
    * 将所有线段合并为一个 Path 进行一次性 `stroke()`；将所有节点/棋子合并后一次性 `fill()`。
3.  **DPR 控制**：强制限制移动端的 `devicePixelRatio` 上限为 **2.0**，平衡视觉清晰度与渲染速度。
4.  **调度节流**：使用 `requestAnimationFrame` 封装 `resize` 和 `render` 事件，并设置像素变化阈值（例如尺寸变化小于 2px 时不执行重绘）。

### C. 数据复用优化
1.  **直接读取边集合**：修改渲染器，使其直接从 `GameEngine` 提供的 `snapshot.edges` 中读取已计算好的连接关系，禁止在渲染器中重新推导连线。

## 5. 交付内容
