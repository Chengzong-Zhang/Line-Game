# Prompt: 15x15 棋盘前端渲染终极性能优化（物理双层画布与 GC 降级）

## 1. 现状背景
目前我已经实现了 `rAF` 锁、脏检查和批量绘制，解决了一大半的性能问题。但是在移动端（高 DPR 环境）运行 15x15 大尺寸棋盘时，依然存在微小的掉帧和卡顿。
经过分析，当前的瓶颈已不再是规则计算，而是集中在 **内存带宽、像素填充率（Fill Rate）压力** 以及 **JS 垃圾回收（GC）瞬时压力**。

## 2. 优化限制（红线）
- **绝对禁止修改游戏规则**：不得触碰 `GameEngine.js` 里的任何领土计算、胜负判定或合法性逻辑。
- 保持渲染层的数据获取方式只读（通过 `snapshot`）。

## 3. 必须执行的核心优化任务

### A. 彻底重构为“物理双层 Canvas” (DOM 结构重构)
- **当前痛点**：现在的离屏缓存机制，每帧都在通过 `ctx.drawImage` 将底层静态缓存拷贝到主画布。在移动端高分辨率下，这种高频的全尺寸位图拷贝（Blit）会让 GPU 不堪重负。
- **优化方案**：
  1. 放弃单 Canvas + 离屏缓存的架构。
  2. 在 `index.html` 中使用**两个绝对定位重叠的 `<canvas>` 元素**。
  3. **底层 Canvas（静态层）**：仅负责绘制背景、棋盘骨架网格。只在 `resize` 或 `gridSize` 发生改变时才执行一次绘制。
  4. **顶层 Canvas（动态层）**：设置 `pointer-events: none`（事件穿透）且背景透明。仅负责绘制棋子、合法落点和最后一步光圈。每次 `render()` 只需 `clearRect` 顶层画布，然后绘制寥寥几十个动态元素。

### B. 消除 Map 与字符串分配造成的 GC 掉帧
- **当前痛点**：在遍历计算渲染数据（如 `_collectBoardData`）时，频繁使用了类似 `${point[0]},${point[1]}` 这样的字符串拼接作为 `Map` 的键名。15x15 的高频遍历会产生大量临时字符串对象，引发垃圾回收（GC）导致瞬时卡顿。
- **优化方案**：彻底移除针对点状态的 `Map` 结构维护。`snapshot.boardMatrix` 本身已经是结构清晰的二维数组，请**直接使用 $O(1)$ 的二维数组索引访问**（如 `boardMatrix[y][x]`）来获取点状态。

### C. 彻底移除 `shadowBlur` 阴影渲染
- **当前痛点**：Canvas 2D 的 `shadowColor` 和 `shadowBlur` 在移动端浏览器的底层实现效率极低，是众所周知的性能杀手。
- **优化方案**：在绘制“最后一步光圈”等特效时，严禁使用原生 shadow API。请将其替换为：**绘制一个半径稍大、透明度较低的实心圆**（例如 `rgba(..., 0.3)`），在视觉上模拟光晕效果即可。

### D. 降级高频小图形绘制
- **当前痛点**：完美的圆形（`ctx.arc`）在进行 GPU 光栅化时开销较大，特别是如果棋盘初始状态存在近百个“合法落点提示点”时。
- **优化方案**：将合法落点提示的小圆点，降级替换为**正方形或菱形**（使用 `moveTo / lineTo / rect`），大幅降低路径点计算成本。

请直接提供重构后的 `Renderer.js` 代码框架，以及与之匹配的 `index.html` 和 `styles.css` 修改方案，以实现真正的双层 DOM Canvas 分离。
