# AI Coding 使用总结

## 1. 使用了哪些 AI 工具

本项目主要使用 Claude Code（Anthropic）辅助开发。我的目标是在不修改原有成熟代码的前提下，为 Python + Pygame 三角网格连线棋增加 Minimax + Alpha-Beta AI，因此让 AI 以阅读代码、提出方案和生成子类扩展代码为主，最终设计取舍与验收由我完成。

## 2. AI 帮你完成了哪些工作

Claude Code 首先分析了现有架构，识别出 `_is_legal_move`、`_add_node`、`_get_adjacent_positions` 等可复用方法，并据此生成 Minimax + Alpha-Beta 搜索框架。它还发现原有 `_compute_inner_hull` 计算过慢，不适合在搜索树中反复调用，转而设计了基于 BFS 的快速领土近似评估函数。为提高剪枝效率，AI 生成了攻击走法优先的排序逻辑；在界面部分，则通过 Pygame 子类加入模式选择、AI 可视化圆圈和后台搜索线程。

## 3. 哪些部分 AI 做得不好，需要你自己判断、修改或调试

AI 初始评估函数偏保守，对攻击威胁权重不足，我通过实际对局重新调整了各子项权重。状态快照最初漏掉 `history_hashes`，导致 Superko 规则在搜索中失效，这也是我检查规则一致性后补上的。搜索深度与响应时间不能只看算法效果，最终由我改成可配置难度参数。界面调试时还发现 `draw()` override 与原方法重复调用 `pygame.display.flip()`，造成叠加层时序问题。线程代码也需要人工修正，例如谨慎安排 `ai_move_result` 置为 `None` 的时机。

## 4. 你对“AI 辅助开发游戏”的体会

AI 很适合快速理解代码、搭建算法骨架和列出优化方向，但它不了解本游戏特有的三点限制、Superko 和保护区规则，必须在提示词中精确说明。它也不能代替真实对局测试，尤其是策略强弱、线程安全和 Pygame 渲染管线问题，仍需要我逐项验证和调试。

## 5. 如果重做，你会如何更好地使用 AI

如果重做，我会先整理规则说明、状态字段清单和性能预算，再让 AI 分阶段实现：先验证合法走法与状态恢复，再测试评估函数，最后接入线程和界面。同时增加针对 Superko、保护区、快照恢复及渲染时序的自动化测试，减少问题集中到集成阶段才暴露。
